# Be-REAL Carbon Decision OS 개발 로드맵

## 1. 문서 목적

이 문서는 Be-REAL Carbon Decision OS를 현재의 분석형 ESG 대시보드에서 기업용 탄소 의사결정 및 실행 지원 시스템으로 발전시키기 위한 개발 우선순위를 정리한 문서이다.

문서 목적은 다음과 같다.

- 현재 제품 상태를 기준으로 현실적인 개발 순서를 정한다.
- 어떤 기능부터 개발해야 완성도가 가장 크게 올라가는지 명확히 한다.
- 기능 추가, 구조 개선, 운영 기능 확장을 단계별로 구분한다.
- 향후 팀 단위 개발이나 발표, 협업 시 공통 기준으로 활용한다.

## 2. 현재 상태 요약

현재 제품은 다음 기능을 이미 갖추고 있다.

- 로그인/회원가입 및 기본 인증
- 프로필 관리
- 회사별 탄소 대시보드
- 경쟁사 비교
- 탄소 비용 시뮬레이터
- 목표관리
- AI 챗봇

현재 제품의 강점은 `분석과 시각화`에 있다. 반면 아직 부족한 부분은 `행동 전환`과 `운영 워크플로우`이다.

현재 한계를 제품 관점에서 정리하면 다음과 같다.

- 사용자가 분석은 할 수 있지만 바로 실행 액션으로 이어지지 않는다.
- 시뮬레이터는 계산 중심이며 추천/결정 흐름이 부족하다.
- 구매 요청, 승인, 실행 상태 추적 기능이 없다.
- 데이터 입력, 리포트, 분석 페이지는 아직 미완성이다.
- 일부 API 연결과 환경 설정은 정리되지 않은 상태다.

즉, 지금 제품은 “좋은 분석 도구”에 가깝고, 목표 상태는 “의사결정과 실행을 돕는 운영 시스템”이다.

## 3. 제품 개발 원칙

앞으로의 개발은 아래 원칙을 따른다.

- 새로운 기능을 많이 붙이기보다, 사용자 흐름을 끊김 없이 연결하는 데 우선순위를 둔다.
- 대시보드보다 시뮬레이터와 실행 워크플로우를 중심으로 제품 가치를 강화한다.
- 실거래 연동보다 먼저 내부 의사결정과 승인 흐름을 완성한다.
- 현재 코드 자산을 최대한 활용하면서 확장 가능한 구조로 정리한다.
- 문서화와 구조 정리를 병행해 개발 방향이 흔들리지 않도록 한다.

## 4. 제품 단계 정의

제품은 아래 4단계로 확장한다.

### Phase 1. 현재 기능 정리 및 제품 완성도 개선

목표:

- 지금 있는 기능들을 더 안정적이고 일관된 제품 흐름으로 정리한다.

핵심 방향:

- 기능 간 연결성 개선
- 하드코딩 및 임시 처리 제거
- 준비 중 화면의 역할 재정의
- 시뮬레이터를 제품의 중심 기능으로 재정리

### Phase 2. 의사결정 지원 기능 강화

목표:

- 단순 계산 결과가 아니라 전략 추천과 비교를 제공하는 단계로 발전시킨다.

핵심 방향:

- 전략 추천 카드
- 추천 사유 설명
- 구매 요청 생성 기능 도입

### Phase 3. 운영 워크플로우 도입

목표:

- 내부 승인과 요청 이력을 포함한 운영형 시스템으로 확장한다.

핵심 방향:

- 구매 요청 목록
- 승인 상태 관리
- 요청 이력 및 감사 로그
- 역할 기반 워크플로우

### Phase 4. 실행 및 거래 연계 확장

목표:

- 장기적으로 실제 배출권 구매 실행까지 연결할 수 있는 구조를 만든다.

핵심 방향:

- 주문 모델
- 실행 상태 관리
- 보유 포지션 추적
- 외부 거래 연동 검토

## 5. 우선순위 기준

기능 우선순위는 아래 기준으로 판단한다.

### P0

- 제품 핵심 가치에 직접 연결되는 기능
- 현재 사용자 흐름의 가장 큰 단절을 해결하는 기능
- 구조적으로 나중 기능의 기반이 되는 작업

### P1

- 제품 경험을 크게 개선하지만, P0 없이도 일부 사용은 가능한 기능
- 운영 완성도를 높이는 기능

### P2

- 있으면 좋지만 핵심 흐름을 막지는 않는 기능
- 확장형 기능

## 6. Phase 1 로드맵

### 6.1 목표

- 현재 구현된 기능을 실제 제품처럼 더 일관되고 안정적으로 만든다.

### 6.2 주요 작업

#### P0. 시뮬레이터 중심 재정리

설명:

- 현재 제품에서 가장 가치가 큰 영역은 시뮬레이터다.
- 향후 추천, 구매 요청, 승인 흐름도 모두 이 화면에서 시작될 가능성이 높다.

작업 항목:

- 시뮬레이터 입력값 구조 정리
- 계산 결과 영역 재구성
- 비용, 리스크, 예산, 해외 노출 결과 구역 명확화
- 대시보드/비교/목표관리와의 연결 문구 정리

기대 효과:

- 제품의 중심이 명확해진다.
- 이후 추천 전략 기능을 붙이기 쉬워진다.

#### P0. 환경 설정 및 연결 정리

설명:

- 현재 일부 API 주소 및 fallback 동작이 혼재되어 있어 제품 완성도를 떨어뜨린다.

작업 항목:

- 하드코딩된 API 주소 제거
- 공통 API config 정리
- 에러/로딩 처리 일관화
- mock fallback 전략 명확화

기대 효과:

- 개발 및 배포 안정성이 올라간다.

#### P1. Data Input 역할 재정리

설명:

- 현재는 화면만 있고 실제 기능이 없다.

작업 항목:

- 단기적으로는 “준비 중”인지 “입력 기능”인지 역할 명확화
- 저장 API 설계 초안 작성
- 추후 연결 가능한 데이터 모델 정의

기대 효과:

- 미완성 화면이 제품 인상을 해치지 않게 된다.

#### P1. Reports / Analytics 재정의

설명:

- 현재는 준비 중 화면이지만, 장기적으로는 제품의 중요한 출력 채널이 될 수 있다.

작업 항목:

- Reports를 “경영진 보고서 생성” 기능으로 재정의
- Analytics를 “심화 분석” 기능으로 재정의
- 각 화면의 목적을 문서화

## 7. Phase 2 로드맵

### 7.1 목표

- 시뮬레이터를 계산기에서 의사결정 워크스페이스로 전환한다.

### 7.2 주요 작업

#### P0. 추천 전략 카드 추가

설명:

- 사용자는 수치만 보는 것보다 선택 가능한 전략안이 제시될 때 더 빠르게 의사결정을 내릴 수 있다.

작업 항목:

- 전략 3종 기본 템플릿 설계
- 예: 즉시 매수형, 분할 매수형, 예산 방어형
- 각 전략에 예상 비용, 리스크, 추천 사유 표시
- 추천안 강조 UI 추가

기대 효과:

- 제품이 “분석 도구”에서 “추천 시스템”으로 한 단계 올라간다.

#### P0. AI 기반 전략 설명 연결

설명:

- 추천안이 있어도 이유가 명확하지 않으면 신뢰도가 낮다.

작업 항목:

- 시뮬레이션 결과를 기반으로 전략 설명 생성
- 추천 이유, 리스크, 유의사항 요약
- 챗봇과 시뮬레이터 전략 설명 연결

기대 효과:

- 사용자가 추천안을 더 쉽게 이해하고 설득할 수 있다.

#### P0. 구매 요청 생성 기능 도입

설명:

- 이 기능이 생기면 비로소 분석 결과가 행동으로 전환된다.

작업 항목:

- 선택 전략에서 구매 요청 생성 버튼 추가
- 요청 데이터 구조 정의
- 요청 생성 API 설계 및 화면 연결

기대 효과:

- 제품의 핵심 가치가 크게 올라간다.

#### P1. 목표관리와 전략 연계

설명:

- 단기 구매 전략이 장기 감축 목표와 얼마나 정렬되는지 보여줄 필요가 있다.

작업 항목:

- 전략별 목표 정렬도 표시
- “단기 비용 최적”과 “장기 목표 정렬”의 trade-off 표현

## 8. Phase 3 로드맵

### 8.1 목표

- 제품에 운영 워크플로우를 도입한다.

### 8.2 주요 작업

#### P0. 구매 요청 목록 및 상세 화면

작업 항목:

- 요청 목록 화면
- 요청 상세 화면
- 상태, 비용, 전략, 생성일, 승인자 정보 표시

#### P0. 승인 상태 모델 도입

작업 항목:

- 상태 정의: 초안, 제출, 승인, 반려, 실행 대기, 완료
- 상태 전환 규칙 정의
- 승인/반려 사유 기록

#### P1. 역할 기반 권한

작업 항목:

- 요청 생성자
- 승인자
- 조회 전용 사용자

#### P1. 감사 로그

작업 항목:

- 누가 언제 어떤 요청을 만들고 수정했는지 기록

기대 효과:

- 제품이 조직 내 협업 도구로 발전할 수 있다.

## 9. Phase 4 로드맵

### 9.1 목표

- 향후 실제 거래 및 실행 관리로 확장 가능한 구조를 만든다.

### 9.2 주요 작업

#### P1. 주문 도메인 설계

작업 항목:

- 주문 생성
- 주문 상태
- 체결 이력

#### P1. 포지션 추적

작업 항목:

- 보유 물량
- 평균 단가
- 잔여 노출량

#### P2. 외부 거래 연동 검토

작업 항목:

- 거래소 또는 브로커 API 조사
- 실거래 연동 구조 초안 설계

주의:

- 이 단계는 제품의 운영 흐름이 안정된 이후에 접근하는 것이 적절하다.

## 12. 리스크 및 대응

### 리스크 1. 기능은 많아지는데 흐름은 더 복잡해질 수 있음

대응:

- 모든 기능을 시뮬레이터 중심 흐름에 연결한다.

### 리스크 2. AI 기능이 제품 중심이 아니라 보조 기능에 머무를 수 있음

대응:

- 챗봇보다 “전략 설명”과 “요청 생성 보조”에 먼저 연결한다.

### 리스크 3. 거래 기능을 너무 빨리 붙이면 구조가 무거워질 수 있음

대응:

- 실거래보다 구매 요청과 승인 흐름을 먼저 만든다.
